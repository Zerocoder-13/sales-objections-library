# Поддержка репозитория «Библиотека возражений и кейсов для B2B-продаж»

## Структура репозитория

| Файл/папка | Назначение |
|---|---|
| `sales-objections-library.md` | **Источник истины** — таблица возражений в Obsidian (13 строк: №, Возражение, Контекст, Ответ/аргумент, Кейс/цифра, Источник) |
| `objections-library-viewer.html` | Самодостаточный HTML/CSS/JS веб-интерфейс над таблицей (массив `DATA` внутри файла) |
| `research-queue.md` | Очередь открытых тем для ресёрча со статусами и архивом выполненных |
| `cases/` | Развёрнутые кейс-файлы (формат — скилл `client-case-story`) |
| `templates/` | Шаблон новой строки таблицы + чек-лист |
| `screenshots/` | Скриншоты веб-интерфейса для USER-GUIDE.md |
| `reports/` | Отчёты по проекту (ресёрч, функциональные тесты) |
| `README.md` | Описание проекта для GitHub (ценность, скриншоты, структура) |
| `USER-GUIDE.md` | Подробная инструкция пользователя со скриншотами |
| `_not-in-repo/` | Файлы, не относящиеся к проекту — в `.gitignore`, не публикуются |

## Запуск веб-интерфейса локально

`objections-library-viewer.html` не открывается напрямую через `file://` в Playwright/некоторых браузерах (CSP/permissions). Поднимай локальный сервер из корня репозитория:

```bash
python -m http.server 8743
```

Затем открывай `http://localhost:8743/objections-library-viewer.html`.

## Внесение изменений в библиотеку

1. **Главное правило**: ресёрч новых кейсов/источников запускается **только по явному запросу менеджера** — никогда автоматически, по расписанию или в фоне.
2. Новая строка добавляется в `sales-objections-library.md` **только после подтверждения** менеджером формулировки (возражение → контекст → ответ → кейс → источник).
3. Если у строки появляется развёрнутый кейс-файл — добавь его в `cases/` (формат — скилл `client-case-story`) и сослись на него из колонки «Источник».
4. После изменения строк в таблице — синхронизируй массив `DATA` в `objections-library-viewer.html` (поля `num, objection, context, argument, caseText, sources, companies, caseFile`).
5. Закрытые темы ресёрча переноси из «Активные задачи» в «Архив выполненных» в `research-queue.md` с датой, результатом и ссылкой на источник.

## Naming-конвенции

- Файлы и папки — только на латинице, английские слова (не транслит)
- Формат: `[дата]-[тема]-[тип].md` (например `2026-06-15-invitro-case-study.md`)
- `README.md`, `USER-GUIDE.md`, `CLAUDE.md`, `SKILL.md` — стандартные имена, без даты

## Обновление README и USER-GUIDE

Обновляй, когда меняется состав/интерфейс проекта (новые функции вьювера, изменение структуры папок, изменение количества строк в таблице):

- **README.md** — держи в синхроне: число строк в таблице, таблицу «До/После», структуру папок, список технологий
- **USER-GUIDE.md** — держи в синхроне с реальным UI вьювера; если меняется интерфейс (новые кнопки, поля формы) — переснимай актуальные скриншоты в `screenshots/` (см. раздел про запуск сервера) и обновляй ссылки на них
- Название проекта везде — **«Библиотека возражений и кейсов для B2B-продаж»**

## Проверка перед коммитом

- [ ] `sales-objections-library.md`: таблица не сломана (количество `|` одинаково в каждой строке), нет оставшихся `*заполнить*` без согласования с менеджером
- [ ] `objections-library-viewer.html`: массив `DATA` — валидный JS (без висячих запятых, кавычки экранированы), страница открывается без ошибок в консоли (кроме безвредного 404 favicon)
- [ ] Ссылки в колонке «Источник» и в `cases/*.md` рабочие (https-ссылки или явное «личный опыт менеджера»)
- [ ] README.md и USER-GUIDE.md соответствуют текущему состоянию проекта
- [ ] `.playwright-mcp/`, `_not-in-repo/`, `.claude/settings.local.json` не попадают в git (проверка через `git status` — не должны быть в staged)

## Публикация изменений в Git

```bash
git status
git diff
git add <конкретные файлы>
git commit -m "..."
git push
```

- Коммитить и пушить — **только после подтверждения менеджера**, как и любое изменение, влияющее на общий репозиторий
- Создавай новые коммиты, не используй `--amend` без явной просьбы
